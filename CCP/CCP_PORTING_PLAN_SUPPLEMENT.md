### 3. RS-485 방향 제어 방식 구체화 미흡 🔴

**현황**
Phase 3의 3-7 항목이 "GPIO로 RTS 핀 제어"로만 기술되어 구체성 부족.

**보완 필요 사항**

RS-485 반이중 방향 제어 방식 중 하나를 선택 및 명시:

| 방식 | 설명 | 조건 |
|------|------|------|
| **커널 내장 RS-485** | `TIOCSRS485` ioctl (SER_RS485_ENABLED 플래그) | i.MX6 UART 드라이버 지원 여부 확인 필요 |
| **GPIO 수동 제어** | 송신 전 GPIO(DE/RE 핀) High, 완료 후 Low | 드라이버 미지원 시 사용 |
| **RTS 자동 제어** | `termios` CRTSCTS 플래그 | 하드웨어 흐름 제어 핀이 DE에 연결된 경우 |

추가 명시 사항:
- RS-485 DE 핀에 연결된 i.MX6 GPIO 번호 (회로도 `HW/아이텍/CCP-V1.4` 참조 필요)
- 바이트 전송 완료 후 DE 핀 Low 전환 타이밍 (tcdrain() 완료 후)
- 반이중 타이밍 마진 값 (예: 1ms 대기)

```c
// 권장 구현 예시 (커널 내장 방식)
struct serial_rs485 rs485conf = {
    .flags = SER_RS485_ENABLED | SER_RS485_RTS_ON_SEND,
    .delay_rts_before_send = 0,
    .delay_rts_after_send = 1,
};
ioctl(fd, TIOCSRS485, &rs485conf);
```

### 4. 문자 인코딩 처리 (WCHAR → char) 🟡

> ⚠️ 우선순위 변경: 🔴 필수 → 🟡 권고 (덴버 프로젝트 — 영문 전용)

**현황**
WinCE는 UTF-16LE(TCHAR/WCHAR)를 사용하며, Linux에서는 `char`(ASCII/UTF-8)를 사용.
덴버 RTD 프로젝트는 **영문 전용**이므로 한글 인코딩 이슈는 해당 없음.

**영향 범위 (축소)**

| 데이터 | 인코딩 상황 | 영향도 |
|--------|-----------|--------|
| `DVDB.dat` 역명/노선명 | 영문 (예: "Union Station", "Peoria") | ASCII 범위, 변환 단순 |
| `HELP.TXT` | 영문 | ASCII 범위, 변환 단순 |
| `password.ini`, `CONFIG.INI` | 숫자/영문 | 영향 없음 |
| 런타임 텍스트 (열차번호, 메시지) | 숫자/영문 | 영향 없음 |

**필요 작업 (단순화)**
- 코드 내 `TCHAR`/`WCHAR`/`_T()` 매크로 → `char`/문자열 리터럴로 일괄 치환
- `_tcscpy`, `_tcslen`, `_stprintf` 등 → `strcpy`, `strlen`, `sprintf`로 변환
- `CString` → `char[]` 또는 `std::string`으로 변환
- 멀티바이트 처리(`mbstowcs`, UTF-8 시퀀스 등)는 **불필요**
- `DVDB.dat`, `HELP.TXT`는 영문이므로 별도 인코딩 변환 도구(`iconv`) 불필요

Phase 2 체크 항목:
```
- [ ] TCHAR/WCHAR → char 일괄 치환 완료 확인
- [ ] _T() 매크로 제거 확인
- [ ] CString → char[]/std::string 변환 확인
```

### 6. 실시간성 요구사항 미검토 🟡

**현황**
열차 제어 시스템이나 실시간 응답에 대한 요구사항 및 OS 설정 방안이 없음.

**보완 필요 사항**

시리얼 수신 스레드 스케줄링 정책 명시:
```c
// 권장 설정
struct sched_param param = { .sched_priority = 50 };
pthread_setschedparam(serial_thread, SCHED_FIFO, &param);
```

Linux 커널 설정 확인:
- `CONFIG_PREEMPT_RT` 패치 적용 여부 (Yocto BSP에 포함 여부 확인)
- 적용 불가 시 `CONFIG_PREEMPT` (선점형 커널) 최소 요구

타이머 정밀도:
- `SetTimer` 100ms 주기 → `timer_create(CLOCK_MONOTONIC, ...)` 으로 대체 시 jitter 허용 범위 명시

### 11. 메모리/자원 관리 전략 없음 ℹ️

**현황**
임베디드 시스템에서 장시간 운행 시 메모리 누수가 심각한 문제가 될 수 있으나 계획 없음.

**권고 사항**
- 동적 할당 최소화 원칙: UI 위젯, DB 결과셋 등 수명이 명확한 객체만 `malloc/free`
- 정적 배열 우선 사용 (스택 또는 전역 배열)
- 개발 단계: `valgrind --leak-check=full` 또는 `-fsanitize=address` 로 누수 점검
- 릴리즈 빌드에서 주기적 메모리 사용량 로깅 (`/proc/self/status` VmRSS 읽기)

### 12. PAC 통신 에러 처리 상세 부족 ℹ️

**현황**
리스크 테이블에 "시리얼 타이밍 차이"만 있고, 통신 에러 발생 시 UI/상태 처리 방안 없음.

**보완 필요 사항**

| 에러 상황 | 현재 처리 | 권고 처리 |
|----------|----------|----------|
| PAC 타임아웃 (N초 이상 수신 없음) | 미정 | 상태 LED 황색/적색 전환, 타임아웃 횟수 카운트 |
| CRC 오류 패킷 수신 | 미정 | 패킷 폐기, 오류 카운터 증가, 로그 기록 |
| 시리얼 포트 오픈 실패 | 미정 | 재시도 N회 후 에러 화면 표시 |
| CNCS 송신 실패 | 미정 | 재시도 횟수/간격 정책 명시 |

타임아웃 기준값 (원본 코드 `Serial.cpp` 분석 후 확인):
- PAC 정상 송신 주기 확인 → 타임아웃 임계값 = 정상 주기 × N 배로 설정
